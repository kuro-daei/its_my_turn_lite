# 11. ブランチ戦略

複数人で同じリポジトリを触るとき、「誰がどのファイルを編集しているか」を管理しないと衝突が起きます。
この章では `apple` リポジトリにルールを追加し、実際にブランチを使った作業フローを体験します。

**Claude は起動したまま、この章の実践を進めてください。**

## 1. CLAUDE.md にルールを追加する

まず、チームの約束事を CLAUDE.md に書いておきましょう。
Claude が自動的にこのルールを読んで、誤った操作を防いでくれます。

```
> CLAUDE.md に「main ブランチには直接コミットしない。作業は必ずブランチを作ってから行う」というルールを追加して
```

Claude が CLAUDE.md を更新したら確認します。

```
> CLAUDE.md の内容を見せて
```

ルールが追加されていれば OK です。これ以降、Claude は main への直接コミットを提案しなくなります。

## 2. ブランチとは

### コミットの流れとブランチの関係

これまでのコミットは、一本の線として積み重なっていました。

```
● ─── ● ─── ● ─── ●   ← main
A     B     C     D
```

ブランチとは、この線から「分岐した作業用のコピー」です。
分岐先で自由に変更を重ねても、元の `main` には影響しません。

```
● ─── ● ─── ● ─── ●          ← main（触らない）
              │
              └── ● ─── ●     ← feature/update-overview（ここで作業）
                   E     F
```

### ブランチがないと何が起きるか

3 人が `main` に直接 push したとします。

```
Aさん ──push──▶ main  ← notes/overview.md を更新
Bさん ──push──▶ main  ← Aさんの変更を上書きしてしまった！
Cさん ──push──▶ main  ← Bさんが消した内容をさらに上書き…
```

- 誰かの変更が知らないうちに消える
- 「いつ・誰が・何を変えたか」が追いにくい
- バグが混入しても原因が分からない

### ブランチを使うとどう変わるか

各自が独立したブランチで作業し、完成したものだけ `main` に取り込みます。

```
          ┌── feature/aさん-overview ──●──●──┐
          │                                   ▼
main ─────●                             マージ ──▶ main
          │                                   ▲
          └── feature/bさん-products ──●──●──┘
```

- `main` は常に「動く・読める」安全な状態を保てる
- 各ブランチの変更は PR でレビューしてからマージする
- 問題があってもそのブランチを捨てるだけで `main` は影響を受けない

### HEAD とブランチの関係

`HEAD` は「今自分がいる場所」を指すポインタです。
ブランチを切り替えると `HEAD` も移動します。

```
$ git switch feature/update-overview
→ HEAD が feature/update-overview を指す

$ git switch main
→ HEAD が main を指す（ファイルも main 時点の内容に戻る）
```

## 3. GitHub Flow を体験する

### GitHub Flow とは

`main` は常に安全な状態に保ち、作業は必ず専用ブランチで行うシンプルなルールです。

```
main ──────────────────────────────── (安全な状態を維持)
         └── feature/作業名 ─────┐
                 (作業ブランチ)   │
                                  ▼
                            PR → マージ → main に統合
```

### ブランチ命名規則

| プレフィックス | 用途 | 例 |
|---|---|---|
| `feature/` | 新コンテンツの追加 | `feature/add-products-info` |
| `fix/` | 誤字・誤記の修正 | `fix/overview-typo` |
| `docs/` | 既存ドキュメントの更新 | `docs/overview-update` |

### 実践：ファイルをブランチで更新する

チームで `apple/notes/` のファイルを分担して更新します。

#### ステップ 1：main を最新にしてブランチを作る

**Aさんの場合（企業概要を更新）:**

```
> main を最新にして、feature/update-overview ブランチを作って
```

**Bさんの場合（製品情報を更新）:**

```
> main を最新にして、feature/update-products ブランチを作って
```

#### ステップ 2：ファイルを編集する

**Aさんの場合:**

```
> notes/overview.md の「売上高」を最新の数字に更新して
```

**Bさんの場合:**

```
> notes/products.md に Vision Pro の情報を追記して
```

#### ステップ 3：コミットして push する

```
> 変更をコミットして push して
```

> [!NOTE]
> ブランチ名が `feature/〇〇` になっているか確認しましょう。
> もし `main` と表示されたら、ステップ 1 からやり直してください。

push が完了したら、GitHub のリポジトリページを開いて自分のブランチが表示されているか確認しましょう。
PR の作成・レビュー・マージは次章で学びます。
