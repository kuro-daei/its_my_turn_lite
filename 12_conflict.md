## 1. コンフリクトとは何か

### コンフリクトが起きる状況

AさんとBさんが **同じファイルの同じ行** を別々のブランチで編集すると、Git はどちらの変更を採用すればよいか判断できなくなります。これが **コンフリクト** です。

```
main ─────── [初期] ──────────────────── マージ時に衝突！
               │                              ↑
               ├── feature/a-edits ── [Aさんの変更]
               │
               └── feature/b-edits ── [Bさんの変更]
```

### コンフリクトマーカー

コンフリクトが起きると、Git はファイルの中に **コンフリクトマーカー** を挿入します。

```
<<<<<<< HEAD
Aさんの変更内容
=======
Bさんの変更内容
>>>>>>> feature/b-edits
```

| マーカー | 意味 |
|---|---|
| `<<<<<<< HEAD` | ここから現在のブランチ（HEAD）の内容 |
| `=======` | 区切り線 |
| `>>>>>>> feature/b-edits` | ここまでマージしようとしているブランチの内容 |

### コンフリクトの解決とは

人間（またはAI）がどちらの変更を採用するかを判断し、マーカーを取り除いて正しい内容に書き直すことです。

## 3. コンフリクトを意図的に起こす

### シナリオ

AさんとBさんが **同じ `apple/info/overview.md`** の「CEO」の情報を別々のブランチで編集します。

- **Aさん**: `feature/a-edits` ブランチで CEO 名を「Tim Cook」と記載
- **Bさん**: `feature/b-edits` ブランチで CEO 名を「Timothy D. Cook」と記載

### Step 1：Aさんのブランチを作って編集する

**Claude への指示（Aさんが実行）:**

```
> main を最新にして、feature/a-edits ブランチを作って
```

```
> info/overview.md の CEO の行を「- CEO: Tim Cook」に変更して、
  コミットして。メッセージは「feat: CEO 名を追加（Aさん）」で
```

```
> push して
```

### Step 2：Bさんのブランチを作って編集する

**Claude への指示（Bさんが実行）:**

```
> main を最新にして、feature/b-edits ブランチを作って
```

```
> info/overview.md の CEO の行を「- CEO: Timothy D. Cook」に変更して、
  コミットして。メッセージは「feat: CEO 正式名を追加（Bさん）」で
```

```
> push して
```

### Step 3：Aさんの PR をマージする

まずAさんの PR をマージします。

```
> feature/a-edits の PR を作ってマージして
```

これで `main` には「Tim Cook」が入りました。

### Step 4：Bさんの PR をマージしようとするとコンフリクト発生

```
> feature/b-edits の PR をマージして
```

GitHub がコンフリクトを検出します。

```
This branch has conflicts that must be resolved.
Conflicting files:
  info/overview.md
```

コンフリクトが起きました。次のステップで解決します。

## 4. コンフリクトを解決する

### Step 1：コンフリクトの状態を確認する

```
> コンフリクトが起きているファイルを見せて
```

Claude がコンフリクトマーカーを含んだファイルを表示します。

```
<<<<<<< HEAD
- CEO: Tim Cook
=======
- CEO: Timothy D. Cook
>>>>>>> feature/b-edits
```

### Step 2：コンフリクトを解決する

どちらの内容を残すかを Claude に指示します。

**両方を残したい場合:**

```
> AさんとBさんの変更を両方残す形でコンフリクトを解決して。
  「- CEO: Tim Cook（Timothy D. Cook）」という形にして
```

**片方を採用したい場合:**

```
> Bさんの変更（Timothy D. Cook）を採用してコンフリクトを解決して
```

Claude がコンフリクトマーカーを取り除いて、正しい内容に書き直します。

```
info/overview.md を更新しました：

- CEO: Tim Cook（Timothy D. Cook）

コンフリクトが解消されました。
```

### Step 3：解決したらコミットして push する

```
> 解決したらコミットして。メッセージは「fix: CEO 名のコンフリクトを解決」で
```

```
git add info/overview.md
git commit -m "fix: CEO 名のコンフリクトを解決"
✓ コミットしました
```

```
> push して
```

### Step 4：PR をマージする

```
> PR をマージして、ブランチも削除して
```

## 5. まとめ

### コンフリクトを防ぐための工夫

| 工夫 | 説明 |
|---|---|
| ファイルを分担する | 同じファイルを同時に編集しない |
| 作業前に pull する | 常に最新の main から作業開始する |
| 小さい変更を頻繁にマージする | 長期間ブランチを放置しない |
| コミュニケーション | 「このファイルを今触っています」とチームに共有する |

### コンフリクト解決のフロー

```
コンフリクト発生
      │
      ▼
Claude に「コンフリクトが起きているファイルを見せて」と依頼
      │
      ▼
どちらの変更を採用するかを判断
      │
      ▼
Claude に解決方法を指示
（「両方残す形で」「Aさん側を採用して」など）
      │
      ▼
Claude が「解決したらコミットして」で完了
      │
      ▼
push して PR をマージ
```

### Claude への指示まとめ

| ステップ | Claude への指示例 |
|---|---|
| 状態確認 | `> コンフリクトが起きているファイルを見せて` |
| 解決 | `> AさんとBさんの変更を両方残す形でコンフリクトを解決して` |
| コミット | `> 解決したらコミットして` |

コンフリクトはチーム開発では必ず起きます。焦らず Claude と一緒に解決しましょう。
