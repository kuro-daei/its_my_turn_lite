## 1. Claude Code (CLI) とは

Claude Code は、Anthropic が提供するターミナル（コマンドライン）で動く AI コーディングアシスタントです。
ブラウザを開かなくても、作業中のターミナルから直接 AI に相談・作業依頼ができます。

Web 版の Claude.ai と異なり、Claude Code はあなたのプロジェクトフォルダに直接アクセスし、
ファイルの読み書きや Git 操作・シェルコマンドの実行を代わりに行うことができます。
「プログラミングに詳しいペアプログラマー」がいつでも隣にいるようなイメージです。

### CLI 版 vs Web 版 比較表

| 比較項目 | Claude Code (CLI) | Claude.ai (Web) |
|---|---|---|
| 起動場所 | ターミナル（VSCode 統合ターミナル含む） | ブラウザ |
| ファイル操作 | ファイルの読み書き・編集を直接実行できる | 手動でコピー＆ペーストが必要 |
| コマンド実行 | bash/shell コマンドを代わりに実行できる | 不可 |
| プロジェクト文脈 | CLAUDE.md を読み込みプロジェクトを把握 | 毎回手動で説明が必要 |
| Git 連携 | git コマンドを自動実行・PR 作成まで対応 | 不可 |
| MCP ツール連携 | 設定した外部ツール（Notion・Gmail 等）を呼び出せる | 不可（一部 Artifacts で対応） |
| 利用料金の確認 | `/cost` コマンドで即確認 | ダッシュボードで確認 |
| 向いている作業 | コーディング・ファイル整理・Git 操作・自動化 | 質問・文章生成・ブレスト |
| 向いていない作業 | 長文を読み込ませての Q&A（コスト注意） | コードの自動実行・ファイル編集 |
| オフライン動作 | 不可（API 通信が必要） | 不可（ブラウザ通信が必要） |

> **ポイント:** ファイルを直接触る作業や Git 操作は CLI 版が圧倒的に速い。
> 情報収集・アイデア出しは Web 版で十分な場合も多い。用途で使い分けよう。

### Claude Code でできることの全体像

```
┌─────────────────────────────────────────────┐
│              Claude Code でできること          │
├─────────────────────────────────────────────┤
│  ファイル操作                                  │
│    読む・書く・作る・削除・移動                  │
├─────────────────────────────────────────────┤
│  コード作業                                   │
│    実装・デバッグ・リファクタリング・テスト作成   │
├─────────────────────────────────────────────┤
│  Git / GitHub                               │
│    コミット・ブランチ・PR 作成・レビュー対応      │
├─────────────────────────────────────────────┤
│  シェル操作                                   │
│    npm・pip・make など任意のコマンド実行         │
├─────────────────────────────────────────────┤
│  MCP 連携（設定が必要）                        │
│    Web 検索・Notion・GitHub・Slack など         │
└─────────────────────────────────────────────┘
```

## 2. 起動と基本操作

### 2-1. インストール（初回のみ）

Claude Code はネイティブインストーラーで提供されています（2025年以降、npm 版は非推奨）。

```bash
curl -fsSL https://claude.ai/install.sh | bash
```

```bash
# インストール確認
claude --version
```

> **WSL ユーザーへ:** WSL2 の Ubuntu ターミナル上で上記コマンドを実行してください。
> インストール後、ターミナルを再起動するか `source ~/.bashrc` を実行するとコマンドが使えるようになります。

> **注意:** 以前は `npm install -g @anthropic-ai/claude-code` でインストールするドキュメントが多く残っていますが、
> 2025年以降は公式に非推奨となりました。上記の curl インストーラーを使ってください。

### 2-2. 認証（初回のみ）

```bash
# 初回起動時に認証が走る
claude

# → ブラウザが開き、Anthropic アカウントでログインを求められる
# → 認証完了後、ターミナルに戻ると利用開始できる
```

認証に使うアカウントは Claude.ai（Web 版）と同じアカウントでかまいません。
認証情報はローカルに保存されるため、2 回目以降は自動でログインされます。

> **API キーでの認証:** 開発者向けに API キーを使った認証方法もあります。
> `ANTHROPIC_API_KEY` 環境変数を設定しておくと、ブラウザ認証をスキップできます。
> 初心者は通常のブラウザ認証で始めるのが簡単です。

### 2-3. 起動方法

```bash
# プロジェクトフォルダに移動してから起動するのが基本
cd ~/projects/my-app
claude

# 起動すると対話モードになる
# > と表示されたら入力受け付け状態
```

起動直後の画面イメージ：

```
╭─────────────────────────────────────╮
│ Claude Code  v1.x.x                 │
│ ✓ Loaded CLAUDE.md                  │
╰─────────────────────────────────────╯

> （ここに質問や指示を入力）
```

`✓ Loaded CLAUDE.md` と表示されていれば、プロジェクトの設定が読み込まれています。
表示されない場合は、CLAUDE.md がまだ作成されていないか、フォルダの場所が違う可能性があります。

### 2-4. 終了方法

```bash
# スラッシュコマンドで終了
/exit

# または Ctrl + C でも終了できる
# または Ctrl + D（EOF）でも終了できる
```

### 2-5. ワンショットモード（対話なし）

毎回対話モードに入らず、1 つの指示だけ実行させたい場合：

```bash
# -p フラグで指示を渡す（スクリプト組み込み向け）
claude -p "このフォルダのファイル一覧を教えて"

# 出力形式を指定することもできる
claude -p "README.md を要約して" --output-format text

# JSON 形式で受け取る（自動化スクリプト向け）
claude -p "package.json のバージョンを教えて" --output-format json
```

> **活用場面:** シェルスクリプトや CI/CD パイプラインに組み込むと、
> 「コミット前に変更内容をチェックする」などの自動化が可能です。

## 3. 基本的な使い方（実例）

対話モードを起動したら、普通の日本語で話しかけるだけです。
「AI に詳しい命令構文が必要」というイメージがあるかもしれませんが、自然な言葉で大丈夫です。

### 実例 1 — ファイルの内容を調べてもらう

```
> README.md の内容を要約して
```

Claude がファイルを読み込み、内容を日本語で要約してくれます。
ファイルパスを指定しなくても、カレントディレクトリを自動的に探してくれます。

### 実例 2 — Markdown ファイルを作ってもらう

```
> 今日調べた「Git のブランチとは」についての調査メモを
  Markdown ファイルで作って。ファイル名は research_branch.md にして
```

Claude がファイルを作成し、Markdown 形式で内容を書いてくれます。
実行前に「このファイルを作成しますか？」と確認を求めるケースもあります。

### 実例 3 — コミットメッセージを提案してもらう

```
> 今の変更内容を確認して、適切なコミットメッセージを提案して
```

Claude が `git diff` で変更内容を確認し、Conventional Commits 形式のメッセージを提案してくれます。
CLAUDE.md にルールを書いておくと、そのルールに沿ったメッセージを自動生成します。

### 実例 4 — リポジトリの構成を把握してもらう

```
> このリポジトリのフォルダ構成を確認して、
  どんな役割のファイルがあるか説明して
```

Claude がフォルダを走査して構成を把握し、説明してくれます。
クローンしたばかりのリポジトリを素早く理解したいときに便利です。

### 実例 5 — Git の状態を確認・説明してもらう

```
> git の状態を確認して、変更があれば内容を説明して
```

Claude が `git status` や `git diff` を実行し、結果を解釈して教えてくれます。
コマンドの出力が読みにくくても、Claude が平易な言葉で説明してくれます。

### 実例 6 — .gitignore を説明してもらう

```
> .gitignore の内容を読んで、
  どんなファイルが Git 管理から除外されているか説明して
```

Claude がファイルを読んで、除外設定の意味を日本語で説明してくれます。
「このファイルも除外すべきか」と相談することもできます。

### 実例 7 — コミット履歴を整理して教えてもらう

```
> 直近 5 件のコミット履歴を見やすく表示して、
  どんな変更が行われてきたか説明して
```

Claude が `git log` を実行し、履歴の流れを要約して報告します。

> **注意:** Claude がファイルを書き換えたりコマンドを実行する際、
> 危険度に応じて確認ダイアログが表示されます。
> 内容をよく確認してから `y`（はい）か `n`（いいえ）を選択してください。
> 「何をするか理解できない」場合は必ず `n` を押して確認しましょう。

## 4. よく使うスラッシュコマンド

対話中に `/` で始まるコマンドを入力すると、特殊な操作ができます。
これらは Claude への「指示」ではなく、ツール自体の操作コマンドです。

| コマンド | 説明 | 使いどころ |
|---|---|---|
| `/help` | 使えるコマンド一覧を表示 | 「何があるか忘れた」ときに |
| `/clear` | 会話履歴をリセット | 別のトピックに切り替えるとき |
| `/exit` | Claude Code を終了 | 作業完了時 |
| `/model` | 使用する AI モデルを切り替え | より高性能 or 安価なモデルに変えたいとき |
| `/cost` | 現在のセッションのコスト（推定）を表示 | 使いすぎが心配なとき |
| `/status` | Claude Code の状態・設定確認 | 設定を確認したいとき |
| `/compact` | 会話履歴を要約して圧縮（コスト削減） | 長時間作業でコンテキストが溢れそうなとき |
| `/init` | プロジェクト用の CLAUDE.md を自動生成 | 新規プロジェクトのセットアップ時 |
| `/doctor` | 環境の問題をチェック・診断 | 動作がおかしいと感じたとき |
| `/permissions` | ファイル操作の許可設定を確認 | 権限まわりのトラブル時 |

### 使用例

```
# 会話をリセットしてから新しいトピックへ
> /clear
✓ 会話履歴をクリアしました

# 現在のコストを確認
> /cost
このセッションのコスト: $0.023（約 3.5 円）
入力トークン: 12,450  /  出力トークン: 3,210

# モデルを切り替え
> /model
? 使用するモデルを選択してください:
  ❯ claude-sonnet-4-5（バランス型・推奨）
    claude-opus-4（高性能・高コスト）
    claude-haiku-3-5（高速・低コスト）
```

### モデルの選び方

| モデル | 特徴 | 向いている用途 |
|---|---|---|
| claude-opus-4 | 最高性能・高コスト | 複雑な設計・難しいバグ修正 |
| claude-sonnet-4-5 | バランス型（デフォルト） | 日常的なコーディング全般 |
| claude-haiku-3-5 | 高速・低コスト | 簡単な質問・ファイル確認 |

> **コスト管理のコツ:** `/cost` を定期的に確認する習慣をつけましょう。
> 長い会話は `/compact` でコンテキストを圧縮すると費用を抑えられます。
> 作業の種類によって `/model` でモデルを使い分けるのも効果的です。

## 5. プランモード（Plan Mode）

### 5-1. プランモードとは

通常モードでは Claude は「考えながら即実行」します。
プランモード（Plan Mode）は **「実行前に計画を立てて、人間に確認してから動く」** モードです。

複雑な作業や、間違えると困るファイル操作を依頼するときに使います。

**プランモードを使うべきシーン:**

- 複数ファイルにまたがる大きな変更をする前
- 本番環境に近いファイルを操作するとき
- 手順が複雑で、途中で方向を変えたいかもしれないとき
- 「何をするか先に確認したい」というとき
- 初めてのプロジェクトで Claude に大きな作業をさせるとき

**通常モードとプランモードの違い:**

| 項目 | 通常モード | プランモード |
|---|---|---|
| 計画の提示 | なし（即実行） | あり（確認後に実行） |
| ファイル変更 | 都度確認ダイアログ | 計画承認後にまとめて実行 |
| 向いている作業 | 小さな修正・質問 | 大規模な変更・複数ファイル操作 |
| 安全性 | 普通 | 高い |

### 5-2. 起動方法

```bash
# プランモードで起動
claude --plan

# または対話中に Shift + Tab を押してトグル切り替え
```

対話モード起動後に切り替える場合：

```
# 対話中に Shift + Tab を押すと表示が変わる
[Plan Mode ON]
> （ここに指示を入力）

# もう一度 Shift + Tab を押すと通常モードに戻る
[Plan Mode OFF]
> （ここに指示を入力）
```

### 5-3. 実際の操作フロー（ASCII 図）

```
┌─────────────────────────────────────────────────────┐
│                   プランモードのフロー                    │
└─────────────────────────────────────────────────────┘

  ユーザー入力
      │
      ▼
┌─────────────┐
│  計画フェーズ  │  Claude が「何をするか」を日本語で列挙
│             │  ファイルは一切触らない
└──────┬──────┘
       │
       ▼ 計画を表示（例）
  ┌──────────────────────────────────────┐
  │ 以下の手順で進めます：               │
  │   1. src/index.js を読み込む         │
  │   2. エラー箇所を特定する            │
  │   3. 修正案を提示する               │
  │   4. 承認後にファイルを書き換える    │
  │                                      │
  │  [y] 実行する  [n] キャンセル        │
  └──────────────────────────────────────┘
       │
       ├─── n → 中止（ファイルは変更なし）
       │         └→ 「別の方法で進めたい」と伝えて再計画も可能
       │
       ▼ y を入力
┌──────────────┐
│  実行フェーズ  │  計画に沿ってステップを順番に実行
│              │  各ステップで進捗を表示
└──────┬───────┘
       │
       ▼
  作業完了レポートを表示
  （変更したファイル一覧・実行したコマンド一覧）
```

### 5-4. プランモードの確認例

```
> package.json の依存関係を最新版に更新して

[計画]
1. package.json を読み込んで依存関係一覧を確認します
2. npm outdated を実行して更新可能なパッケージを特定します
3. 各パッケージの最新バージョンと変更内容を確認します
4. package.json を更新します
5. npm install を実行して依存関係をインストールします
6. npm test を実行して動作を確認します

上記の手順で進めてよいですか？ [y/n]
```

`n` を押して別の方法を指定することもできます：

```
n

> テストは実行しなくていい。4 まで完了したら止まって確認させて
```

> **初心者へのアドバイス:** 最初はプランモードを積極的に使いましょう。
> 「Claude が何をしようとしているか」を事前に確認する習慣が、
> 意図しない変更を防ぐ一番の安全策です。
> 慣れてきたら通常モードと使い分けるようにしていきましょう。

## 6. CLAUDE.md の役割と書き方

### 6-1. CLAUDE.md とは

CLAUDE.md は、Claude Code に「このプロジェクトのルールや文脈」を伝えるための設定ファイルです。

Claude Code を起動すると、プロジェクトフォルダにある `CLAUDE.md` を自動的に読み込みます。
これにより、毎回「このプロジェクトは〇〇です」と説明しなくて済みます。

**CLAUDE.md がない場合 vs ある場合の違い:**

| 状況 | CLAUDE.md なし | CLAUDE.md あり |
|---|---|---|
| プロジェクト理解 | 毎回説明が必要 | 起動時に自動把握 |
| コミットメッセージ | Claude が自由に生成 | ルールに従って生成 |
| コーディング規約 | Claude が推測して実装 | 指定した規約で実装 |
| 注意事項 | Claude は知らない | 事前に周知できる |

**CLAUDE.md に書くと効果的な内容:**

- プロジェクトの概要・目的
- 使用している技術スタック（言語・フレームワーク・ライブラリ）
- フォルダ構成の説明
- コーディング規約（命名規則・スタイルガイド）
- コミットメッセージのルール
- よく使うコマンド（起動・テスト・ビルドなど）
- 注意事項・やってはいけないこと

### 6-2. 配置場所

```
my-project/           ← プロジェクトルート
├── CLAUDE.md         ← ここに置く（プロジェクト用）
├── src/
│   └── index.js
└── package.json
```

また、ホームディレクトリに置くとすべてのプロジェクトで共通して読み込まれます：

```
~/.claude/            ← ユーザー共通設定
└── CLAUDE.md         ← すべてのプロジェクトに適用される
```

サブフォルダに置くことも可能で、そのフォルダ内の作業時に読み込まれます：

```
my-project/
├── CLAUDE.md              ← プロジェクト全体のルール
├── src/
│   ├── CLAUDE.md          ← src/ 配下のみに適用されるルール
│   └── index.js
└── docs/
    ├── CLAUDE.md          ← docs/ 配下のみに適用されるルール
    └── README.md
```

> **優先順位:** プロジェクトの `CLAUDE.md` はユーザー共通の設定を上書きします。
> 「プロジェクト固有のルール」をプロジェクト側に、
> 「自分の作業スタイル」をユーザー側に書き分けるのがおすすめです。

### 6-3. CLAUDE.md の書き方例

```markdown
# CLAUDE.md

## プロジェクト概要

このリポジトリは ○○ サービスのフロントエンドです。
React 18 + TypeScript + Vite で構成されています。
対象ユーザー: 社内の営業チーム（非エンジニア）

## フォルダ構成

src/
├── components/   UIコンポーネント（再利用可能な部品）
├── pages/        ページ単位のコンポーネント
├── hooks/        カスタムフック
├── utils/        ユーティリティ関数
└── types/        TypeScript 型定義

## よく使うコマンド

- 開発サーバー起動: `npm run dev`
- テスト実行: `npm test`
- ビルド: `npm run build`
- リント: `npm run lint`
- 型チェック: `npm run typecheck`

## コーディング規約

- コンポーネント名はパスカルケース（例: UserCard）
- 関数名・変数名はキャメルケース（例: getUserData）
- コメントは日本語で書く
- `any` 型は使用しない
- Props には必ず型定義をつける

## コミットメッセージルール

Conventional Commits 形式を使用：
- feat: 新機能
- fix: バグ修正
- docs: ドキュメント
- refactor: リファクタリング
- test: テスト追加・修正

## 注意事項

- .env ファイルは絶対にコミットしない
- main ブランチへの直接プッシュは禁止
- API キーはコードに直書きしない
- PR レビューなしでマージしない
```

### 6-4. CLAUDE.md を自動生成する

新規プロジェクトでは、Claude Code 自身に雛形を作ってもらえます：

```
> /init
```

Claude がプロジェクトのフォルダ構成や package.json などを分析し、
CLAUDE.md の初稿を自動生成します。生成後に内容を確認し、不足している情報を追記しましょう。

### 6-5. CLAUDE.md のメンテナンス

CLAUDE.md は一度書いたら終わりではなく、プロジェクトの変化に合わせて更新します。

```
> CLAUDE.md を見直して、最新の状況と合っていない部分を教えて
```

と依頼すると、Claude が現在の CLAUDE.md とプロジェクト実態を比較して、
更新が必要な箇所を提案してくれます。

### Claude Code を使いこなすための 5 つのコツ

1. **CLAUDE.md を最初に整備する**
   プロジェクトの文脈を書いておくと、毎回説明しなくて済む。`/init` で自動生成も可能。
   プロジェクトが変化したら定期的に更新することも忘れずに。

2. **プランモードを積極的に使う**
   大きな変更の前は必ず `--plan` か `Shift + Tab` でプランモードに切り替えて確認する。
   初心者のうちは「全作業をプランモードで」というくらいでちょうどよい。

3. **`/cost` でコストを定期確認する**
   長時間の作業後は必ず確認。予算オーバーを防ぐために `/compact` も活用する。
   モデルを使い分けることでコストをコントロールできる。

4. **MCP は少しずつ追加する**
   最初は WebSearch/WebFetch だけ設定し、慣れてから他のツールを追加する。
   設定ファイルに API キーを直書きしないことを徹底する。

5. **Git 操作は Claude に任せる**
   コミットメッセージの生成・PR 作成まで Claude に任せると作業効率が大きく上がる。
   ただし、Claude が生成した内容は必ず自分で確認してから実行・マージする。

### よくある失敗と対処法

| 失敗パターン | 対処法 |
|---|---|
| Claude がプロジェクトを誤解する | CLAUDE.md に詳細を記載 / `/clear` して文脈をリセット |
| コストが予想以上にかかる | `/compact` で圧縮 / 安価なモデルに切り替え |
| 意図しないファイルが変更された | `git diff` で確認して `git restore` で戻す |
| MCP が接続できない | `/status` で状態確認 / API キーと設定ファイルを再確認 |
| 英語で回答が返ってくる | 「日本語で答えてください」と指示 / CLAUDE.md に言語設定を追記 |
