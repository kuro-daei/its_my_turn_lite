# 補足 04. マージとコンフリクト解決

> **位置づけ:** この補足資料は本編の発展内容です。[09_pull_request.md](../../09_pull_request.md) を学んでから読むことを推奨します。


コードを複数人で編集していると、いつか必ず「コンフリクト（競合）」という壁にぶつかります。
コンフリクトは怖いものではありません。Git が「どちらの変更を使うか、自分では判断できなかった」と正直に教えてくれているだけです。
この章では、コンフリクトの仕組みを理解し、落ち着いて解決できるようになることを目標にします。

## 1. コンフリクトが起きる原因

### 1-1. 基本的な仕組み

Git のマージは、2 つのブランチの変更を自動的に統合しようとします。
**別々の行**を変更した場合は自動マージに成功しますが、**同じ行**を異なる内容に変更した場合は Git が判断できずにコンフリクトが発生します。

### 1-2. 典型的なシナリオ（ASCII 図）

`main` ブランチと `feature/greeting` ブランチで、2 人が同じファイルの同じ行を変更した場合を見てみましょう。

```
【共通の出発点 (base)】
hello.txt の 1 行目:
  "Hello, World!"

        ┌─────────────────────────────────────┐
        │ 共通の祖先コミット (base commit)     │
        │  hello.txt: "Hello, World!"          │
        └──────────────┬──────────────────────┘
                       │
          ┌────────────┴────────────┐
          │                         │
          ▼                         ▼
┌─────────────────────┐   ┌─────────────────────┐
│  main ブランチ       │   │ feature/greeting     │
│  (Alice の変更)      │   │ (Bob の変更)          │
│                     │   │                     │
│  hello.txt:         │   │  hello.txt:         │
│  "Hello, Alice!"    │   │  "Hello, Bob!"      │
└─────────────────────┘   └─────────────────────┘
          │                         │
          └────────────┬────────────┘
                       │
                       ▼
              ┌────────────────┐
              │  マージしようとする  │
              │                │
              │  ??? どちらを   │
              │  使えばよい？   │
              │                │
              │  CONFLICT!     │
              └────────────────┘
```

ポイントは次のとおりです。

- **異なる行を変更した場合**: Git が自動マージ → コンフリクトなし
- **同じ行を異なる内容に変更した場合**: Git が判断できない → コンフリクト発生

### 1-3. コンフリクトが発生するタイミング

コンフリクトは以下の操作で発生します。

| 操作 | コンフリクトの可能性 |
|---|---|
| `git merge` | あり（ブランチ統合時） |
| `git pull` | あり（リモートの変更を取り込む際） |
| `git rebase` | あり（コミット履歴の整理時） |
| `git cherry-pick` | あり（特定コミットの適用時） |

## 2. コンフリクトマーカーの読み方

コンフリクトが発生すると、Git は該当ファイルに**コンフリクトマーカー**と呼ばれる特殊な記号を書き込みます。

### 2-1. マーカーの構造

```
<<<<<<< HEAD
Hello, Alice!
=======
Hello, Bob!
>>>>>>> feature/greeting
```

各マーカーの意味は次のとおりです。

| マーカー | 意味 |
|---|---|
| `<<<<<<< HEAD` | 現在いるブランチ（マージ先）の変更の開始 |
| `=======` | 2 つの変更の境界線 |
| `>>>>>>> feature/greeting` | マージしようとしたブランチの変更の終了 |

つまり：

- `<<<<<<< HEAD` ～ `=======` の間 → **自分のブランチ（現在のブランチ）の変更**
- `=======` ～ `>>>>>>> feature/greeting` の間 → **相手のブランチの変更**

### 2-2. 実際のファイル全体のイメージ

```
# あいさつプログラム

<<<<<<< HEAD
名前: Alice
メッセージ: Hello, Alice!
=======
名前: Bob
メッセージ: Hello, Bob!
>>>>>>> feature/greeting

以上です。
```

コンフリクトマーカーは「どこで競合しているか」を正確に示しています。
マーカーのある行だけを修正し、マーカー自体（`<<<<<<<`、`=======`、`>>>>>>>`）は必ず削除してください。

### 2-3. コンフリクトマーカーが残ったままにすると

コンフリクトマーカーが残ったままコミットすると、プログラムのソースコードの場合は文法エラーになり、Markdown などのテキストの場合はそのまま文書内に表示されてしまいます。
**必ずマーカーを完全に取り除いてから** `git add` してください。

## 3. VS Code での解決

VS Code はコンフリクトを視覚的に解決するための UI を提供しています。コンフリクトのあるファイルを開くと、マーカー部分が色分けされ、選択ボタンが表示されます。

### 3-1. 画面の見え方

コンフリクトが発生したファイルを VS Code で開くと、競合している箇所に次のようなボタンが表示されます。

```
╔═══════════════════════════════════════════════╗
║  Accept Current Change                        ║  ← 現在のブランチの変更を採用
║  Accept Incoming Change                       ║  ← 相手ブランチの変更を採用
║  Accept Both Changes                          ║  ← 両方の変更を残す
║  Compare Changes                              ║  ← 差分を並べて比較する
╚═══════════════════════════════════════════════╝

  <<<<<<< HEAD（緑色でハイライト）
  Hello, Alice!
  =======
  Hello, Bob!（青色でハイライト）
  >>>>>>> feature/greeting
```

### 3-2. 各ボタンの効果

#### Accept Current Change（現在の変更を採用）

自分のブランチ（`HEAD`）の内容だけを残します。相手の変更は破棄されます。

```
# ボタンを押した後の結果
Hello, Alice!
```

使いどき: 自分の変更が正しく、相手の変更は不要な場合。

#### Accept Incoming Change（入ってくる変更を採用）

マージしようとしたブランチの内容だけを残します。自分の変更は破棄されます。

```
# ボタンを押した後の結果
Hello, Bob!
```

使いどき: 相手の変更が最新・正確で、自分の変更を上書きしてよい場合。

#### Accept Both Changes（両方の変更を採用）

両方の変更を順番に残します（自分の変更が先、相手の変更が後）。

```
# ボタンを押した後の結果
Hello, Alice!
Hello, Bob!
```

使いどき: 両方の内容を合わせて使いたい場合。ただし、追加後に文章の流れを確認・調整が必要な場合がほとんどです。

#### Compare Changes（差分を比較）

2 ペインの差分ビューを開きます。どちらが何を変更したか並べて確認できるので、判断に迷ったときに便利です。

### 3-3. VS Code での解決の流れ（全体）

1. VS Code でコンフリクトファイルを開く（エクスプローラーで `C` マークが付いているファイル）
2. 競合箇所のボタンをクリックして解決方針を選ぶ
3. 必要に応じて手動で文章を調整する
4. ファイルを保存する（`Ctrl + S`）
5. すべてのコンフリクトが解決できたらターミナルへ進む

VS Code の左サイドバーにある「ソース管理」パネルでは、コンフリクトのあるファイルが `C`（Conflicted）として一覧表示されます。すべての `C` がなくなるまで繰り返しましょう。

## 4. コマンドラインでの解決手順

VS Code が使えない環境や、マーカーを直接編集したい場合のコマンドライン手順です。

### 4-1. コンフリクト発生を確認する

```bash
git status
```

出力例:

```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   hello.txt

no changes added to commit (use "git add" and/or "git commit -a")
```

`both modified:` と表示されているファイルがコンフリクトしています。

### 4-2. ファイルを手動で編集する

テキストエディタでコンフリクトファイルを開き、マーカーを含む競合部分を手動で修正します。

修正前（コンフリクト状態）:

```
<<<<<<< HEAD
Hello, Alice!
=======
Hello, Bob!
>>>>>>> feature/greeting
```

修正後（解決済み）:

```
Hello, Alice and Bob!
```

ルール:
- `<<<<<<< HEAD`、`=======`、`>>>>>>> ブランチ名` の 3 行は必ず削除する
- 残す内容は自由（一方だけでも、両方でも、まったく新しい文章でも可）
- ファイルとして文法的・意味的に正しい状態にする

### 4-3. 解決済みとしてマークする

```bash
git add hello.txt
```

コンフリクトを解決したファイルを `git add` することで、「このファイルの競合は解決した」と Git に伝えます。

複数ファイルにコンフリクトがある場合は、すべてのファイルを解決してから `git add` します。

```bash
# 複数ファイルを一度に追加する場合
git add ファイル1 ファイル2 ファイル3
```

### 4-4. マージコミットを作成する

```bash
git commit
```

オプションなしで実行するとエディタが開き、デフォルトのマージコミットメッセージ（`Merge branch 'feature/greeting'` など）が表示されます。
そのまま保存して閉じればコミット完了です。

コミットメッセージを直接指定する場合:

```bash
git commit -m "fix: コンフリクトを解決し feature/greeting をマージ"
```

### 4-5. マージを中断したい場合

解決が難しくなった場合や、状況をリセットしたい場合はマージを中断できます。

```bash
git merge --abort
```

これでマージ前の状態に戻ります。落ち着いてから改めてチャレンジしましょう。

### 4-6. 解決の全体フロー（まとめ）

```
git merge feature/greeting
        │
        ▼
   CONFLICT 発生
        │
        ▼
  git status でコンフリクトファイルを確認
        │
        ▼
  エディタでファイルを手動編集
  （マーカーを削除し、正しい内容にする）
        │
        ▼
  git add <解決済みファイル>
        │
        ▼
  git status で未解決ファイルがないか確認
        │
        ▼
  git commit（マージコミットを作成）
        │
        ▼
      完了!
```

## 5. コンフリクトを防ぐ習慣

コンフリクトを完全になくすことはできませんが、発生頻度を大幅に減らす習慣があります。

### 5-1. こまめに pull する

作業を始める前、また作業中もこまめに `git pull` してリモートの最新状態を取り込みましょう。

```bash
# 作業開始前の習慣
git checkout main
git pull
git checkout feature/自分の作業
git merge main    # または git rebase main
```

**なぜ効果があるか**: コンフリクトは「変更の差分が大きいほど」起きやすくなります。こまめに同期することで差分を小さく保てます。

### 5-2. 小さいブランチ・短命ブランチ

1 つのブランチでの変更を小さく保ち、できるだけ早くマージすることが重要です。

| 良い例 | 避けるべき例 |
|---|---|
| 1 機能 = 1 ブランチ | 何週間も開発して巨大なブランチを作る |
| 数日以内にマージ | 長期間 main と乖離させる |
| 変更ファイルを絞る | 関係ないファイルまで変更する |

**目安**: 1 ブランチの変更は 200〜300 行以内に収めると管理しやすくなります（あくまで目安）。

### 5-3. 作業範囲を分けてもらう

チームで作業する場合は、**担当するファイルや機能を事前に分担**することでコンフリクトを根本的に減らせます。

```
良い分担の例:
  Alice → header.md, about.md を担当
  Bob   → footer.md, contact.md を担当
  ↓
同じファイルを触らないのでコンフリクトが起きにくい
```

作業を始める前に「このファイルを変更する」と宣言しておくと、チーム内での調整がスムーズになります。

### 5-4. コンフリクト防止のチェックリスト

作業開始時:
- [ ] `git pull` で最新状態を取得した
- [ ] ブランチを作成してから変更を始めた
- [ ] 担当範囲（ファイル）をチームと確認した

作業中:
- [ ] 定期的に `git pull` または `git merge main` している
- [ ] 変更量が増えすぎる前に細かくコミットしている

作業完了時:
- [ ] プルリクエストを早めに出した
- [ ] レビュー待ちの間も `main` の変化を追っている

## 6. git stash — 作業を一時退避する

コンフリクトとは直接関係しませんが、マージやブランチ切り替えの前に「作業途中の変更を一時退避したい」場面でよく使うのが `git stash` です。

### 6-1. git stash が必要な場面

```
【よくある状況】

  feature/new-feature ブランチで作業中...
  ファイルを途中まで編集した状態
        │
        ▼
  「急いで別のバグを直してほしい」と頼まれた
        │
        ▼
  git checkout main をしようとすると...
  「変更がコミットされていないので切り替えられない」
  または「変更が次のブランチに持ち越されてしまう」
        │
        ▼
  git stash で作業を一時退避!
  → クリーンな状態でブランチを切り替えられる
```

### 6-2. 基本的な使い方

#### 作業を退避する

```bash
git stash
```

実行例と出力:

```bash
$ git stash
Saved working directory and index state WIP on feature/new-feature: abc1234 前回のコミットメッセージ
```

このコマンドで、コミットしていない変更（ステージング済み・未ステージング両方）が退避されます。
ワーキングディレクトリが直前のコミット状態に戻り、クリーンな状態になります。

コメントをつけて退避する場合（後から区別しやすい）:

```bash
git stash push -m "ログイン機能の途中"
```

#### 退避した作業の一覧を確認する

```bash
git stash list
```

出力例:

```
stash@{0}: WIP on feature/new-feature: abc1234 前回のコミットメッセージ
stash@{1}: On main: ログイン機能の途中
stash@{2}: WIP on feature/old-branch: def5678 別の作業
```

`stash@{0}` が最も新しい退避です。番号は 0 から始まります。

#### 退避した作業を元に戻す（pop）

```bash
git stash pop
```

`stash@{0}`（最新のもの）を取り出し、ワーキングディレクトリに適用します。
適用が成功すると、その stash はリストから削除されます。

特定の stash を指定して取り出す場合:

```bash
git stash pop stash@{1}
```

#### 退避した作業を適用するが、リストには残す（apply）

```bash
git stash apply
```

`pop` との違いは、適用後も `stash list` にエントリが残る点です。
同じ退避を複数のブランチに適用したいときに使います。

### 6-3. git stash の全体フロー（実例）

```bash
# 1. feature/new-feature ブランチで作業中（途中まで編集）

# 2. 急な依頼が来たので作業を退避
git stash push -m "新機能の実装途中"

# 3. 別ブランチに切り替えてバグ修正
git checkout main
git checkout -b fix/urgent-bug
# ... バグ修正 ...
git add bugfix.txt
git commit -m "fix: 緊急バグを修正"
git checkout main
git merge fix/urgent-bug

# 4. 元の作業ブランチに戻る
git checkout feature/new-feature

# 5. 退避していた作業を復元
git stash pop

# 6. 作業を再開!
```

### 6-4. stash に関するその他のコマンド

特定の stash の内容を確認する:

```bash
git stash show stash@{0}
# 変更されたファイルの一覧表示

git stash show -p stash@{0}
# 変更内容の差分を表示（-p は --patch の略）
```

stash を削除する（適用せずに捨てる）:

```bash
git stash drop stash@{0}
# 特定の stash を削除

git stash clear
# すべての stash を削除（元に戻せないので注意）
```

### 6-5. stash のよくある落とし穴

**新規ファイルが退避されない場合**

デフォルトの `git stash` は、`git add` していない新規ファイルを退避しません。

```bash
# 新規ファイルも含めて退避する場合
git stash push --include-untracked
# または
git stash push -u
```

**stash を pop したときにコンフリクトが発生する場合**

stash を pop しようとしたブランチで、退避中に同じ箇所が変更されていると、pop 時にコンフリクトが発生します。
この場合は通常のコンフリクト解決手順（Section 4 参照）で対処します。
`git stash drop stash@{0}` は自動では行われないので、解決後に手動で削除が必要です。

---

## まとめ

| 場面 | 使うコマンド |
|---|---|
| コンフリクト発生後に状況確認 | `git status` |
| VS Code で視覚的に解決 | エディタのボタンをクリック |
| コマンドラインで解決 | 編集 → `git add` → `git commit` |
| マージを中断して最初からやり直す | `git merge --abort` |
| 作業を一時退避 | `git stash` |
| 退避した作業を一覧表示 | `git stash list` |
| 退避した作業を復元 | `git stash pop` |

コンフリクトは Git を使いこなすうえで避けられない経験です。最初は戸惑うかもしれませんが、何度か解決するうちに必ず慣れます。
「コンフリクトが発生した = 作業がぶつかった」という事実を冷静に確認し、手順通りに解決しましょう。
