## 1. Pull Request とは

### ブランチの変更を main に取り込む「依頼」

前章でブランチに変更を push しましたが、まだ `main` には反映されていません。

```
GitHub

  main: ●──────────────────────────  ← まだ変わっていない
                  ↑
  feature/update-overview: ●──●──●  ← ここに変更が入っている
```

PR とは「このブランチの変更を `main` に取り込んでいいですか？」という **依頼 + 議論の場** です。

```
GitHub

  feature/update-overview ──[PR]──▶ main へのマージ依頼
                                          │
                                    チームが差分を確認
                                    コメント・承認
                                          │
                                          ▼
                                    マージ → main に統合
```

### なぜ直接 main にマージしないのか

PR を経由することで：

| 直接 push | PR 経由 |
|---|---|
| 誰も確認しない | チームが変更を確認できる |
| 問題があっても気づかない | コメントで指摘・修正できる |
| 「なぜこの変更か」が残らない | PR の説明・会話が履歴に残る |

### PR の流れ全体像

```
[Aさん]                          [GitHub]                    [Bさん]

feature/ ブランチを push
        │
        ▼
PR を作成 ──────────────────▶ PR がオープン
                                    │
                                    │ ◀──────── PR の内容を確認
                                    │ ◀──────── コメント・承認
                                    │
                             マージ（main に統合）
                                    │
        ◀──────────────────── git pull で最新を取得
```

## 2. PR を作成する

前章で push したブランチに対して、各自が PR を作ります。

**Aさんの場合（企業概要を更新）:**

```
> PR を作って。タイトルは「feat: Apple 企業概要を更新」で
```

**Bさんの場合（製品情報を更新）:**

```
> PR を作って。タイトルは「feat: 製品情報に Vision Pro を追記」で
```

Claude が PR を作成します。

```
PR を作成しました：
タイトル: feat: Apple 企業概要を更新
ベース:   main ← feature/update-overview
URL:      https://github.com/[ユーザー名]/apple/pull/1
```

PR の URL をブラウザで開いて、変更内容が正しく表示されているか確認しましょう。

```
> PR をブラウザで開いて
```

## 3. お互いの PR をレビューする

チームメンバーの PR を確認し合います。

### PR 一覧を確認する

```
> PR の一覧を見せて
```

```
#1  feat: Apple 企業概要を更新     [feature/update-overview]  - Aさん
#2  feat: 製品情報に Vision Pro 追記 [feature/update-products]  - Bさん
```

### 相手の PR の内容を確認する

```
> PR #2 の内容を見せて
```

```
title:  feat: 製品情報に Vision Pro 追記
state:  OPEN
author: Bさん

変更内容:
  notes/products.md — 3 lines added
```

差分も確認できます。

```
> PR #2 の差分を見せて
```

### コメントを残す

問題なければ承認コメントを残します。

```
> PR #2 に「内容確認しました。LGTM!」とコメントして
```

修正が必要な場合はコメントで指摘します。

```
> PR #2 に「Vision Pro の価格が古い情報です。最新版に更新をお願いします」とコメントして
```

指摘を受けた側は修正して追加コミットします。

```
> notes/products.md の価格を修正して、コミットして push して
```

push すると自動的に同じ PR に追加されます。

---

## 4. PR をマージする

チームの確認が取れたらマージします。

```
> PR #1 をマージして、ブランチを削除して
```

```
✓ PR #1 をマージしました
✓ ブランチ feature/update-overview を削除しました
```

マージ後は全員が `main` を最新にします。

```
> main を最新にして
```

```
【マージ後の状態】

GitHub (main)                        全員のローカル (main)
  ●──────────────────────●    ◀──── git pull ────   ●
                          ↑                          notes/overview.md ★
                  PR #1 マージ済み                   notes/products.md ★
  feature/update-overview: 削除済み
```

---

## 5. まとめ

| ステップ | Claude への指示 |
|---|---|
| PR 作成 | `> push して PR を作って` |
| PR 確認 | `> PR の一覧を見せて` / `> PR #1 の内容を見せて` |
| 差分確認 | `> PR #1 の差分を見せて` |
| コメント | `> PR #1 に「LGTM!」とコメントして` |
| マージ | `> PR #1 をマージして、ブランチを削除して` |
| 同期 | `> main を最新にして` |

PR を使うことで、チームの変更がすべて可視化され、安全に `main` を保てます。
