# 11. Pull Request ワークフロー

Pull Request（PR）は、チームで安全にファイルを変更するための仕組みです。
Claude に依頼するだけで、PR の作成・レビュー・マージまで完結できます。
**Claude は起動したまま、この章の内容を試してください。**

## 1. Pull Request とは

### 1-1. PR の役割

Pull Request とは、あるブランチの変更を別のブランチ（通常は `main`）に取り込むよう依頼する仕組みです。

```
feature/update-overview ── [commit A] ── [commit B]
                                              ↓
                                    PR を作成して main へ取り込み依頼
main ──────────────────────────────────────────────
```

PR を通じて行われる「レビュー」には、以下の目的があります。

| 目的 | 説明 |
|------|------|
| 内容の確認 | 複数の目でファイルを確認することで、見落としを防ぐ |
| 知識の共有 | チームメンバー全員がどんな変更が加わったかを把握できる |
| 品質の維持 | 誤字・情報の正確性・フォーマットの一貫性を確認できる |
| 安全なマージ | main ブランチに直接 push せず、確認済みの内容だけを取り込む |

### 1-2. apple リポジトリでの PR フロー

```
1. feature ブランチを作成して作業
   > feature/update-contacts ブランチを作って

2. 変更を commit して push
   > info/contacts.md をコミットして。メッセージは「feat: 担当者を追加」で
   > push して

3. PR を作成
   > PR を作って。タイトルは「feat: Apple 担当者を追加」で

4. チームメンバーが内容を確認・コメント
   > PR の内容を見せて

5. 修正が必要であれば追加 commit して push

6. 確認が取れたら main にマージ
   > PR をマージして、ブランチも削除して
```

## 2. PR を作成する

### 基本的な PR 作成

ブランチで変更をコミット・push した後、Claude に依頼します。

```
> push して PR を作って
```

タイトルや説明を指定したい場合：

```
> PR を作って。タイトルは「feat: Apple 企業概要を更新」で
```

レビュアーを指定する場合：

```
> PR を作って。タイトルは「feat: 担当者情報を追加」、レビュアーは [GitHubユーザー名] で
```

### ドラフト PR（まだレビュー不要な場合）

作業中であることを明示したい場合は、ドラフト PR を使います。

```
> ドラフト PR を作って。タイトルは「WIP: 打ち合わせメモを作成中」で
```

作業が完了したら通常の PR に変換します。

```
> PR #3 をドラフトから通常の PR に変換して
```

## 3. PR のレビュー（チームで確認し合う）

### PR の内容を確認する

```
> PR の内容を見せて
> PR #3 を見せて
> PR をブラウザで開いて
```

### 差分を確認する

```
> PR の差分を見せて
> PR #3 の差分を見せて
```

### apple/info/ ファイルの PR をレビューする例

Bさんが Aさんの `info/overview.md` 更新 PR をレビューする場合：

```
> PR #2 の内容を見せて
```

```
title:  feat: Apple 企業概要を最新情報に更新
state:  OPEN
author: Aさん

## 変更内容
- 売上高を FY2024 の数値に更新
- CEO 名を Tim Cook に明記

Changes: 1 file changed, 8 insertions(+), 3 deletions(-)
```

差分を確認して問題なければ：

```
> PR #2 に「企業概要の更新を確認しました。数値も正確です。LGTM!」とコメントして
```

変更が必要な場合：

```
> PR #2 に「売上高の単位がドルか円か不明です。追記をお願いします」とコメントして
```

### PR を承認する

```
> PR #2 を承認して
```

## 4. PR をマージする

### マージ方法の種類

PR をマージする方法は 3 種類あります。

#### merge（マージコミット）

```
main:    A --- B --------- M
                \         /
feature:         C --- D
```

feature ブランチのすべての commit が保持され、マージコミット（M）が作られます。

#### squash（スカッシュマージ）

```
main:    A --- B --- S
                      ↑
feature:  C --- D   （C と D をひとつにまとめた S）
```

feature ブランチの複数の commit を 1 つにまとめて main に追加します。

#### rebase（リベースマージ）

```
main:    A --- B --- C' --- D'
                      ↑
feature:  C --- D   （main の先頭に付け直した C', D'）
```

### 推奨マージ方法

| 状況 | 推奨方法 | 理由 |
|------|----------|------|
| 小さな修正・情報追加 | `--squash` | 履歴がシンプルになる |
| 大きな変更（履歴を残したい） | `--merge` | レビューの文脈が残る |

### Claude へのマージ指示

```
> PR をマージして、ブランチも削除して
> PR #2 をスカッシュマージして、ブランチも削除して
> PR #2 をマージして、main を最新にして
```

## 5. PR のベストプラクティス

### 5-1. 小さい PR を作る

大きすぎる PR はレビューが困難になります。目安として 1 PR = 1 つの目的 にしましょう。

**良い例（`apple` リポジトリの場合）:**

```
PR 1: 「feat: Apple 企業概要を追加」（overview.md のみ）
PR 2: 「feat: 担当者情報を追加」（contacts.md のみ）
PR 3: 「feat: 1月打ち合わせメモを追加」（2025-01-meeting.md のみ）
```

### 5-2. わかりやすい説明文の書き方

PR の説明文（本文）には以下を含めると、チームメンバーが判断しやすくなります。

```markdown
## 変更内容
- info/contacts.md に山田さんの連絡先を追加
- 備考欄に打ち合わせ方法の好みを追記

## 変更理由
1月の打ち合わせで担当者情報を取得したため

## 確認してほしい点
- メールアドレスの形式が正しいか
- 備考の内容が適切か
```

### 5-3. PR テンプレートの活用

リポジトリに PR テンプレートを置くと、PR 作成時に本文が自動的に埋まります。

```
> .github/pull_request_template.md を作成して、以下の内容を書いて：

## 変更内容

<!-- 何を変更したか、箇条書きで記載 -->
-

## 変更理由

<!-- なぜこの変更が必要か -->

## 確認してほしい点

<!-- レビュアーに特に確認してほしいポイント -->

## 動作確認

- [ ] ローカルでファイルを確認した
- [ ] Markdown のプレビューで見た目を確認した

> .github/pull_request_template.md をコミットして push して
```

## 6. gh CLI 便利コマンド一覧

> **参考:** このセクションのコマンドはすべて Claude への自然言語指示で実行できます。

### PR 一覧・確認系

| コマンド | 説明 | Claude への指示例 |
|---------|------|-----------------|
| `gh pr list` | 開いている PR の一覧表示 | `> PR の一覧を見せて` |
| `gh pr view` | 現在のブランチの PR を表示 | `> PR の内容を見せて` |
| `gh pr view 3` | 番号で PR を表示 | `> PR #3 を見せて` |
| `gh pr view --web` | ブラウザで PR を開く | `> PR をブラウザで開いて` |
| `gh pr status` | 自分に関連する PR のステータス | `> PR のステータスを確認して` |
| `gh pr diff` | PR の差分を表示 | `> PR の差分を見せて` |

### PR 操作系

| コマンド | 説明 | Claude への指示例 |
|---------|------|-----------------|
| `gh pr create` | PR を作成 | `> push して PR を作って` |
| `gh pr merge --squash -d` | スカッシュマージ・ブランチ削除 | `> PR をスカッシュマージしてブランチも削除して` |
| `gh pr review --approve` | PR を承認 | `> PR #3 を承認して` |
| `gh pr comment` | PR にコメントを追加 | `> PR #3 に「LGTM!」とコメントして` |

---

## まとめ

| ステップ | Claude への指示例 |
|---------|---------|
| PR 作成 | `> push して PR を作って` |
| PR 確認 | `> PR の内容を見せて` |
| 差分確認 | `> PR の差分を見せて` |
| 承認 | `> PR #3 を承認して` |
| コメント | `> PR #3 に「確認しました。LGTM!」とコメントして` |
| マージ | `> PR をスカッシュマージして、ブランチも削除して` |

PR を活用することで、チームでの変更が可視化され、安全に `main` ブランチを保てます。
